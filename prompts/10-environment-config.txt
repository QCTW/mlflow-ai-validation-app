================================================================================
PROMPT 10: ENVIRONMENT CONFIGURATION
================================================================================

OBJECTIVE:
Set up environment variable templates for both backend and frontend

FILE 1: .env.example (Root - Backend Configuration)

PURPOSE: Template for backend environment variables

VARIABLES:

1. PORT=5000
   Purpose: Backend server listening port
   Why: Allow different ports in different environments
   Default: 5000 (standard MLflow port + 1 to avoid conflict)

2. MLFLOW_TRACKING_URI=http://localhost:5000
   Purpose: URL of MLflow tracking server
   Why: Backend needs to know where MLflow is running
   Examples:
   - Development: http://localhost:5000
   - Production: https://mlflow.company.com
   - Remote: http://192.168.1.100:5000

3. FRONTEND_URL=http://localhost:5173
   Purpose: Frontend origin for CORS configuration
   Why: Security - only allow requests from known frontend
   Default: 5173 (Vite default port)

USAGE:
1. Copy: cp .env.example .env
2. Edit .env with actual values
3. .env is gitignored (in .gitignore)
4. .env.example is committed (safe template)


FILE 2: client/.env.example (Frontend Configuration)

PURPOSE: Template for frontend environment variables

VARIABLES:

1. VITE_API_URL=http://localhost:5000/api
   Purpose: Backend API base URL
   Why: Frontend needs to know where to send API requests
   Note: VITE_ prefix required for Vite to expose to browser
   Examples:
   - Development: http://localhost:5000/api
   - Production: https://api.company.com
   - Staging: https://staging-api.company.com

VITE ENVIRONMENT VARIABLES:
- Must start with VITE_ prefix
- Embedded in browser bundle at build time
- Accessible via import.meta.env.VITE_API_URL
- Different from Node.js process.env

USAGE:
1. cd client
2. Copy: cp .env.example .env
3. Edit .env with actual values

SECURITY NOTES:

1. Never Commit .env Files
   - Contains sensitive data
   - Different per environment
   - Already in .gitignore

2. Commit .env.example Files
   - Safe template without secrets
   - Helps new developers
   - Documents required variables

3. Frontend Variables
   - VITE_ variables are PUBLIC
   - Embedded in browser bundle
   - Never put secrets in VITE_ variables
   - Only use for non-sensitive configuration

4. Backend Variables
   - Server-side only
   - Can contain secrets (API keys, database URLs)
   - Never exposed to browser

BENEFITS:

1. Environment Flexibility
   - Same code runs in dev, staging, production
   - Configure via environment variables

2. Security
   - Secrets not in source code
   - Easy to rotate secrets

3. Team Collaboration
   - .env.example documents requirements
   - New developers know what to configure

4. CI/CD Ready
   - Set environment variables in deployment pipeline
   - No code changes needed
